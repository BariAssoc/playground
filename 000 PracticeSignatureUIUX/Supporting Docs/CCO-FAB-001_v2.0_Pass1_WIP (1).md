## §1. Universal FAB Definition (NEW — locks Val's principle)

A **FAB (Focused Action Block)** is any behavior, action, signal, or rhythm placed between a Warm-up Bookend and a Cool-down Bookend, making it deterministically measurable, repeatable, and compoundable into trend.

**The FAB content varies infinitely.** The bookend measurement protocol is fixed. This is what makes FABs behavioral biometrics rather than activity logs.

**Master analogy (locked, unchanged):**

> *FAB is to BEHAVIOR what HRV is to STRESS.*

**Val's principle, May 2, 2026:**

> *"Anything which looks or feels like a FAB should be pouring up as a possible FAB."*

This principle is operationalized in the companion canon **CCO-FAB-CANDIDATE-001** (FAB Candidate Detection Algorithm — always-active).

## §3. Visibility Classification (NEW — restores lost rule)

Every FAB carries one of two visibility classes:

| Class | Definition | Patient Sees? | Examples |
|---|---|---|---|
| **Task FAB** | Patient-actionable, displayed in Journal | ✅ Yes | Protein, Hydration, Sleep, Medication, Hunger, Exercise |
| **Silent FAB** | System-supportive, keeps routine/segment integrity | ❌ No — Ollie reacts behind scenes | Segment-Stabilizer, Erosion-Prevention, Formation-Support, Emergency-FAB |

**This restores the visibility rule that was unclear after FAB Canon v1.1A (Jan 30, 2026).**

The two classes are **orthogonal** to families and operating levels — every FAB has a family, an operating level, AND a visibility class.

## §4. Six States (UNCHANGED from v1.2)

```
Learning → Active → Stable → Drifting → Locked → Blocked
```

**Critical rule:** No "Completed" state. FABs cycle — they never terminate. A FAB can regress from Stable back to Drifting, or from Drifting back to Active.

## §7. Bookend + Pin Architecture (HYBRID — clarified)

**Bookends wrap FABs.** Pins are **sensors inside Bookends.**

### Always-on pins (sensors, NOT FABs)

- **Timestamp** (mandatory, 4 levels: should-start, actual-start, duration, end)
- **Mood**
- **Effort**

### Open Pin Set (extensible, NOT FABs)

- Screen time
- App interaction patterns
- iOS Focus Mode
- Response latency
- Voice tone (proposed)
- (Future pins added as architecture evolves)

### Discrete behaviors → become FABs

- Productivity blocks → **FAB-FOCUS™**
- Hunger logs → **FAB-HUNGER™**
- Naps → **FAB-NAP™**
- Social events → **FAB-SOCIAL-EVENT™**

**Rule:** Pin measurement specs are deferred (TBD — future Pass).

That's really what's important on the bookend so the pin is open as we evolved to name it because we have to define at one point how we measure on every bookend and how you measured the mood and the effort and how do the score."*

The pin set is **open and extensible.** New pins are added as the platform evolves.

## §11. FCS — FAB Consistency Score (UNCHANGED from v1.1A)

```
FCS = (0.6 × Completion Rate) + (0.4 × Timing Accuracy)
Range: 0.0–1.0 over 7/14/30 day windows
Baseline: 14 consecutive days FCS ≥ 0.70 = Learning → Stable
```
